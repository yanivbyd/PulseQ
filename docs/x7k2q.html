How Rate Limiting Works at Scale

PulseQ Daily Brief

If you operate any public API or high-traffic service, you eventually face the same problem: some clients send too many requests. Sometimes it's accidental (a bug or retry storm). Sometimes it's abusive (scrapers or bots). And sometimes it's simply success — your product gets popular faster than your backend can handle.

Rate limiting is the safety valve. It controls how many requests a client can make within a given time window, protecting your system's stability while keeping the experience fair for everyone.

At small scale, rate limiting looks straightforward. At large scale, it becomes a distributed systems problem involving consistency, latency, fairness, and cost.

🎯 The Core Goal

At its heart, rate limiting answers one question:

Should this request be allowed right now?

To answer that, the system must track:

  • Who is making the request (user ID, API key, IP, etc.)
  • How many requests they've made recently
  • What the policy is (for example: 100 requests per minute)

Under the limit → allow. Over the limit → reject (HTTP 429). Simple in theory. The complexity appears when traffic is large, distributed, and bursty.

📍 Where Rate Limiting Happens

1 · Client-side (rarely trusted)

The client limits itself. This reduces load but cannot be trusted for enforcement.

2 · API Gateway (most common)

The gateway sits in front of services and enforces limits centrally.

✅ Pros

  • Protects all downstream services
  • Easy to manage policies
  • Reduces duplicate logic

❌ Cons

  • Can bottleneck if poorly designed

3 · Service-level limiting

Each service enforces its own limits.

✅ Pros

  • Fine-grained control
  • Local decisions

❌ Cons

  • Harder to maintain consistency
  • More duplication

At scale, many companies use gateway-level + service-level defense in depth.

⚡ The Three Classic Algorithms

🪟 Fixed Window Counter

Divide time into fixed windows, count requests, reset when the window rolls over. If a user sends 100 requests at 12:00:59 and another 100 at 12:01:01, they send 200 requests in ~2 seconds — yet stay within the rules.

✅ Pros

  • Very simple
  • Memory efficient
  • Fast to compute

❌ Cons

  • Burst at window boundaries
  • Unfair traffic spikes

📊 Sliding Window Log

Store timestamps of recent requests. On each request: remove entries older than 60 seconds, count remaining, allow or reject.

✅ Pros

  • Very accurate
  • Smooth enforcement
  • Prevents boundary bursts

❌ Cons

  • Memory heavy at scale
  • Expensive to maintain
  • Harder to distribute

🪣 Token Bucket (Most Popular)

Each client has a bucket of tokens added at a fixed rate. Each request consumes one token. No tokens = rejected. Example: 10 tokens/sec, bucket size 100 — burst up to 100 instantly, sustained at 10/sec.

✅ Pros

  • Smooth traffic shaping
  • Supports bursts gracefully
  • Memory efficient
  • Distributed-friendly

❌ Cons

  • More complex to implement
  • Needs time synchronization

🌐 Distributed Challenges

Problem 1 · Multiple Servers

Each node might allow the same user independently, multiplying the limit. Solutions: centralized store (Redis), consistent hashing per user, local + global hybrid limits.

Problem 2 · Latency vs Accuracy

Every request may need a read/write to the rate-limit store. Strong consistency = higher latency. Local caching = risk of over-allowing. Large systems often accept small inaccuracies for much better performance.

Problem 3 · Hot Keys

Popular API keys become hotspots in Redis. Mitigations: key sharding, local pre-checks, hierarchical limiting, probabilistic limiting.

🏭 What Production Systems Do

Real systems layer their defenses:

  1. Edge/CDN limiting — blocks obvious abuse early
  2. Gateway token bucket — primary enforcement
  3. Service-level protection — safety net
  4. Adaptive rules — dynamic tightening during incidents

Additionally: per-user, per-IP, per-endpoint limits, global protection, and burst vs sustained policies.

🎨 Designing Good Rate Limits

Be transparent. Return clear 429 responses with X-RateLimit-Limit, X-RateLimit-Remaining, and Retry-After headers.

Allow reasonable bursts. Users often send traffic in spikes.

Differentiate by user tier. Free vs paid users should have different limits.

Monitor and tune. Rate limits almost always need adjustment in production.

💡 Final Thoughts

Rate limiting starts simple but quickly becomes a distributed systems challenge at scale. The choice of algorithm, storage strategy, and enforcement layer all impact fairness, performance, and reliability.

Most modern systems converge on token bucket–style limiting backed by a fast centralized store like Redis, combined with layered defenses at the edge and service levels.

When implemented well, rate limiting is invisible to good users and highly effective against abuse — exactly what you want from a production-grade system.
