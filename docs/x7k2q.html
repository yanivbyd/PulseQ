How Rate Limiting Works at Scale

PulseQ Daily Brief

If you operate any public API or high-traffic service, you eventually face the same problem: some clients send too many requests. Sometimes it's accidental (a bug or retry storm). Sometimes it's abusive (scrapers or bots). And sometimes it's simply success — your product gets popular faster than your backend can handle.

Rate limiting is the safety valve. It controls how many requests a client can make within a given time window, protecting your system's stability while keeping the experience fair for everyone.

At small scale, rate limiting looks straightforward. At large scale, it becomes a distributed systems problem involving consistency, latency, fairness, and cost. Let's walk through how it actually works in production systems.

The Core Goal of Rate Limiting

At its heart, rate limiting answers one question:

Should this request be allowed right now?

To answer that, the system must track:

  • Who is making the request (user ID, API key, IP, etc.)
  • How many requests they've made recently
  • What the policy is (for example: 100 requests per minute)

If the client is under the limit → allow the request.
If over the limit → reject (usually HTTP 429).

Simple in theory. The complexity appears when traffic is large, distributed, and bursty.

Where Rate Limiting Happens

Before diving into algorithms, it's important to understand placement. Rate limiting can live in several layers:

1. Client-side (rarely trusted)

The client limits itself. This reduces load but cannot be trusted for enforcement.

2. API Gateway (most common)

This is the typical production choice. The gateway sits in front of services and enforces limits centrally.

Pros

  • Protects all downstream services
  • Easy to manage policies
  • Reduces duplicate logic

Cons

  • Can become a bottleneck if poorly designed

3. Service-level limiting

Each service enforces its own limits.

Pros

  • Fine-grained control
  • Local decisions

Cons

  • Harder to maintain consistency
  • More duplication

At scale, many companies use gateway-level + service-level defense in depth.

The Three Classic Algorithms

Most production systems use one of three main approaches.

Fixed Window Counter

Divide time into fixed windows (e.g., per minute), count requests within the window, and reset when the window rolls over.

If a user sends 100 requests at 12:00:59 and another 100 at 12:01:01, they effectively send 200 requests within ~2 seconds — yet still remain within the rules.

Pros

  • Very simple
  • Memory efficient
  • Fast to compute

Cons

  • Allows burstiness at window boundaries
  • Can create unfair traffic spikes

Sliding Window Log

Instead of fixed buckets, the system stores timestamps of recent requests and checks how many occurred within the last N seconds. On each request: remove timestamps older than 60 seconds, count remaining, allow or reject.

Pros

  • Very accurate
  • Smooth enforcement
  • Prevents boundary bursts

Cons

  • Memory heavy at scale
  • More expensive to maintain
  • Harder to distribute

Token Bucket (Most Popular at Scale)

Each client has a bucket of tokens. Tokens are added at a fixed rate (e.g., 10 per second). Each request consumes one token. If no tokens remain, the request is rejected. The bucket has a maximum capacity.

Example: fill rate 10 tokens/sec, bucket size 100. The client can burst up to 100 requests instantly, but sustained traffic is limited to 10/sec.

Pros

  • Smooth traffic shaping
  • Supports bursts gracefully
  • Memory efficient
  • Works well in distributed systems

Cons

  • Slightly more complex to implement
  • Requires careful time synchronization

Distributed Rate Limiting Challenges

At scale, the real difficulty is not the algorithm — it's distribution.

Problem 1: Multiple Servers

If you run many gateway instances, each node might allow the same user independently, effectively multiplying the limit. Common solutions: centralized store (Redis is very common), consistent hashing per user, local + global hybrid limits.

Problem 2: Latency vs Accuracy

Every request may need a read/write to the rate-limit store. Strong consistency means higher latency; local caching risks over-allowing. Large systems often accept small inaccuracies for much better performance.

Problem 3: Hot Keys

Popular API keys can become hotspots in Redis or other stores. Mitigations include key sharding, local pre-checks, hierarchical rate limiting, and probabilistic limiting.

What Production Systems Actually Do

In real high-scale systems, rate limiting is usually layered:

  1. Edge/CDN limiting — blocks obvious abuse early
  2. Gateway token bucket — primary enforcement
  3. Service-level protection — safety net
  4. Adaptive rules — dynamic tightening during incidents

Additionally, mature systems include per-user limits, per-IP limits, per-endpoint limits, global system protection, and burst vs sustained policies.

Designing Good Rate Limits

Good rate limiting is not just technical — it's product design.

Be transparent. Return clear 429 responses and include headers like X-RateLimit-Limit, X-RateLimit-Remaining, and Retry-After.

Allow reasonable bursts. Users often send traffic in spikes.

Differentiate by user tier. Free vs paid users should have different limits.

Monitor and tune. Rate limits almost always need adjustment in production.

Final Thoughts

Rate limiting starts simple but quickly becomes a distributed systems challenge at scale. The choice of algorithm, storage strategy, and enforcement layer all impact fairness, performance, and reliability.

Most modern systems converge on token bucket–style limiting backed by a fast centralized store like Redis, combined with layered defenses at the edge and service levels.

When implemented well, rate limiting is invisible to good users and highly effective against abuse — exactly what you want from a production-grade system.
